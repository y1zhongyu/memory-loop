# memory-loop

## 定位

AI记忆健康检查与强化工具。核心价值：
1. 诊断现有记忆质量，找出为什么记了还是会犯
2. 帮使用者看清AI的问题模式
3. 有记忆体系的强化，没有的帮忙搭建
4. session内决策锚点，防止长对话漂移

## 与其他记忆工具的分工

**与self-improvement的关系（如果已安装）：**
- self-improvement：记"学到了什么"，包括最佳实践、知识更新、功能请求，范围更广
- memory-loop：专注"犯了什么错+为什么没改好"，提供诊断和修复闭环，深度更深
- 两者不替代，各司其职
- 错误记录有重叠：统一由memory-loop的ERRORS.md管理，self-improvement的`.learnings/ERRORS.md`只记命令失败等技术类错误，行为类错误不重复记录

**与MEMORY.md的关系：**
- MEMORY.md是长期记忆，memory-loop的ERRORS.md是错误清单，两者定位不同
- 初始化时如发现MEMORY.md里有错误记录，汇报给使用者并由使用者决定处理方式（见初始化流程第2步）

## 文件结构

```
memory-loop/
├── SKILL.md        # 使用说明与初始化流程
├── RULES.md        # 评分规则与犯错边界
├── ERRORS.md       # 错误清单（活跃+归档两层）
├── SCOREBOARD.md   # 分数看板
├── DIAGNOSIS.md    # 诊断报告模板
├── REPAIR.md       # 记忆修复流程
└── DECISIONS.md    # session内决策锚点（临时）
```

## 初始化流程

```
1. 询问使用者："是否开始搭建memory-loop？"
   → 否：结束
   → 是：先扫描workspace，预估完成时间后告知使用者，再继续

2. agent主动扫描workspace常见规范文件
   （SOUL.md / TOOLS.md / MEMORY.md / AGENTS.md / 模板文件等）

   扫描三件事：

   A. 检查是否已安装self-improvement
      → 已安装：告知使用者分工规则（见上方"与其他记忆工具的分工"）
      → 未安装：无需处理

   B. 提取现有错误记录、问题清单
      → 扫描完成后，如实告知使用者：发现了什么、在哪个文件、什么格式、共几条
      → 询问：「除此之外还有其他地方也记了错误吗？」
      → 把所有来源汇总后，根据实际情况生成迁移方案供使用者选择：
        · 例：错误分散在多个文件 → 方案A全部迁移统一管理 / 方案B按文件性质分别处理
        · 例：已有专项维护文件 → 方案A业务类留原文件+通用类进ERRORS.md / 方案B全部迁移
        · 方案数量和内容视实际扫描结果而定，不固定，给出推荐理由
      → 按使用者选择的方案执行，完成后按以下分类导入ERRORS.md：
        · 通用行为类：流程、沟通、格式等与具体业务无关的错误
        · 业务内容类：与特定业务场景相关的错误（如游戏分析、代码等）
          → 如果使用者已有专项记录文件，业务类错误继续维护在原文件，ERRORS.md里只记通用行为类

   C. 识别已配置的agent及其任务类型
      → 告知使用者："我识别到以下agent：[列出+任务类型]"
      → 询问："是否都需要接入？有没有要调整的？"
      → 确认后在ERRORS.md和SCOREBOARD.md里为每个agent开section

   如果扫描结果为空（既无错误记录也无agent配置）：
   → 进入【体系搭建模式】：
     a. 询问需要接入哪些agent
     b. 询问这些agent主要做什么类型的任务
     c. 根据任务类型推导常见错误风险点，按通用/业务分类
     d. 生成初始错误清单框架供使用者确认
     e. 生成对应的规范建议

3. 执行初次诊断（见DIAGNOSIS.md）
   → 输出诊断报告

4. 询问是否使用默认评分规则
   → 是：用标准RULES.md / 否：逐项确认

5. 在HEARTBEAT.md里追加以下内容：

   ## memory-loop 周复盘检查
   每次heartbeat执行时：
   - 读取SCOREBOARD.md，检查各agent当前分数
   - **仅当**有agent低于70分时，才向使用者推送：错误清单+问题模式分析
   - 如果当前日期跨入新的一周（周一），触发：
     a. 上周分数归档进SCOREBOARD.md历史区
     b. 各agent分数重置为100
     c. ERRORS.md活跃区中超过30天未触发的条目移入归档区
   - 无异常时：此项检查静默，不向用户汇报

   ## memory-loop 版本检查
   每次heartbeat执行时：
   - 进入 skills/memory-loop/ 目录，执行 `git fetch`
   - 对比本地与远端的 version 字段（读SKILL.md头部）
   - **仅当**远端版本更高时，才提示使用者：「memory-loop 有新版本（当前 X.X.X → 最新 X.X.X），运行 git pull 更新」
   - 已是最新或目录不是git仓库：此项检查静默，不向用户汇报

6. 初始化完成，各agent从本周开始计分
```

## ERRORS.md 分类与管理结构

每个agent的错误记录按以下分类维护，并执行严格的“防膨胀”管理：

```
## [Agent名]

### 【通用行为类 · 活跃区】（物理上限：约5000 tokens）
（流程跳步、沟通缺失、格式违规等与具体业务无关的错误）
⚠️ 防膨胀机制：
- 同类合并：新错误若与已有条目同源，必须在旧条目上"合并"（更新案例和重复次数），绝不新增条目。
- 强制瘦身：若活跃区过度膨胀，主动将最不常犯的低频错误移入归档区。

### 【业务内容类 · 活跃区】
（事实错误、信息遗漏等与特定业务场景相关的错误）
→ 如果使用者已有专项记录文件，此区块留空并注明指向路径

### 【归档区】（超过30天未触发，或因活跃区触顶被淘汰）
⚠️ 极致压缩：移入此区的记录必须压缩为单行文本：
`[日期] [错误类型]: [一句话描述] (重复X次) | [修复动作]`
```

## 日常使用

```
- 每次任务开始前：建议读ERRORS.md活跃区（推荐，非强制）
  → 要强制执行，需在agent自身SOUL.md里加入对应约束

- 跨Session任务缓存：DECISIONS.md不再是单次聊天的临时文件，而是跨Session的任务状态与规范锚点。
  采用【活跃/休眠】双区机制管理：

  ✅ 写入原则（强制结构化与去重）：
  · 写入时机：确认任何格式规范、模板细节、用户偏好、业务参数时立即写入。
  · 无损还原测试：条目必须能100%还原细节（如：不能写"已确认格式"，必须写"标题加粗+尖括号"）。
  · 结构化压缩：只写【对象】+具体规则，禁止背景废话。
  · 强制同类覆盖：新规则直接原地修改/替换旧条目，绝不新增并存条目。
  · 写入位置：所有新增和反复修改的条目均写入 `## [活跃区]`。

  ⚠️ 读取时机（物理强制，禁止盲猜）：
  · 在生成任何多行结构化输出（如报告、列表、模板套用）前，或执行多步骤长任务（如spawn子进程）前。
  · 必须先读取DECISIONS.md的 `[活跃区]`。
  · 绝不允许以"我以为我记得"为由跳过读取。

  🔄 状态流转与淘汰（防止无限膨胀）：
  · 移入休眠：当某个明确的子任务/阶段完成（如"敏捷汇报已确认，进入深度报告阶段"），主动将相关格式规则从 `[活跃区]` 移入 `## [休眠区]`。
  · 物理清理与持久化：当文件内容过多（如超过30条/2000 tokens）触发清理兜底。从最老的 `[休眠区]` 条目开始，有长期价值的转存 `MEMORY.md` 或专项文件，无价值的直接物理删除。绝不主动全量清空文件。

  决策覆盖与渐进式收敛（防漂移）：
  · 同一议题讨论超过3轮，主动提炼「当前共识」请用户确认。
  · 存疑决策（"凑合/先这样"）标注⚠️存疑，不作为强锚点。

- 错误触发机制（按信号强度分级）：

  ✅ 强信号（必须立即记录+扣分，不等下一轮，不等用户再催）：
  · 用户明确指出错误（"这错了"、"不对"、"不是这个意思"）
  · 用户要求重来、推翻输出（"重新做"、"算了我来"）
  · 用户直接忽略输出并重新发需求（沉默式否定）

  ⚠️ 弱信号（记录为"待确认"，本轮主动确认，有回答再决定是否扣分）：
  · 用户表达迟疑（"这不太对吧"、"感觉哪里不对"）
  · 用户接受但有明显保留（"好吧"、"凑合"、"随便"）
  · 用户修改了AI给的内容后直接用（说明AI没做到位）
  → AI主动确认：「我注意到你调整了X，是我理解有偏差吗？」
  → 用户确认是错误 → 触发记录+扣分
  → 用户说是自己改需求 → 不记录

  批量错误拆分规则：
  · 用户说"这几点都有问题"时，逐条列出、逐条确认，逐条分别记录
  · 不允许合并为一条整体记录（否则后续单独犯其中一项时无法触发"重复犯"加倍）

  执行时序（强信号触发时，当轮回复必须包含）：
  1. 承认错误（一句话）
  2. 记录ERRORS.md
  3. 更新SCOREBOARD.md
  4. 输出扣分说明
  → 顺序不可颠倒，不可跨轮延迟

  → 通用行为类错误：记入ERRORS.md
  → 业务内容类错误：记入对应专项文件（如有），同时在ERRORS.md业务区注明

- 错误触发与报警预警（不翻旧账，人设解耦）：
  · 分数变动且低于阈值才推送，每次报警**只推送"自上次报警后新增的错误"**，绝不翻旧账。
  · 生成报警或强提示文案时，必须严格结合 Agent 自身的 `SOUL.md` 和 `USER.md` 生成符合人设和特定称谓的话术，严禁使用硬编码默认词汇（防OOC）。

- 修复回血机制（带伤回血，上限80分）：
  · 分数日常只扣不加，取消所谓的日常微量加分。
  · **唯一合法回血途径**：完整执行 `DIAGNOSIS.md` 和 `REPAIR.md`（并修改了底层规范文件）。
  · 修复完成后，当前分数恢复至**最高 80 分**（如 58分修完回80；85分修完不变）。
  · 经过修复的错误，在活跃区打上 `✅已修复` 标记，后续绝不再计入日常报警。

- 周复盘与算总账（展示原始病情）：
  · 使用者说"给我出本周复盘"时，立即执行DIAGNOSIS.md。
  · 周末出具复盘报告时，除了展示最终得分，**必须单独列出本周原始扣分总计**与修复回血分数，让用户看清AI真实的犯错程度。
```
